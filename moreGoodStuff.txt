Great question — this is a key architectural decision. Short answer: no, you almost certainly should not store full running configs. Here's why and what you do instead.

## What You Store in Git: Desired State, Not Full Configs

A full NX-OS running config can be 2,000–10,000+ lines. Multiply by 1,200 devices and you get:
- Massive, unreadable diffs on every change
- Merge conflicts constantly
- Secrets/hashes leaking into git (enable passwords, SNMP community strings, crypto keys)
- Most of the config is boilerplate or auto-generated (default lines, interface counters, etc.)

### Instead, store intent-based YAML — only the things you actively manage:

# configs/nxos/dc1-leaf-01.yml
hostname: dc1-leaf-01
role: leaf
site: dc1
vlans:
  - id: 10
    name: MGMT
  - id: 100
    name: PROD_WEB
interfaces:
  - name: Ethernet1/1
    description: uplink-to-spine1
    mode: trunk
    allowed_vlans: "10,100"
  - name: Ethernet1/47
    description: server-port
    mode: access
    access_vlan: 100
bgp:
  asn: 65001
  router_id: 10.0.1.1
  neighbors:
    - address: 10.0.0.1
      remote_as: 65000
ntp_servers:
  - 10.10.10.1
  - 10.10.10.2

This is maybe 30–50 lines per device instead of thousands. At 1,200 devices, that's still manageable.

## Layered Config Model (Critical at Scale)

You don't want 1,200 separate files that each repeat the same NTP servers and syslog config. Use a hierarchy:

configs/
  global.yml              # NTP, DNS, syslog, AAA — applies to ALL devices
  roles/
    leaf.yml              # Common leaf switch config
    spine.yml             # Common spine config
    ios-access.yml        # Common IOS access layer config
  sites/
    dc1.yml               # Site-specific overrides (TACACS server, etc.)
    dc2.yml
  devices/
    dc1-leaf-01.yml       # Only device-unique values (hostname, IPs, BGP neighbors)
    dc1-leaf-02.yml

Ansible merges these layers at runtime. A device-specific file might only be 10–15 lines — just hostname, loopback IP, and BGP peer addresses. Everything else is inherited.

## Where Full Configs Live

Full running configs do get stored, just not in your source-of-truth repo:

- AAP/Ansible backups: A scheduled job runs nxos_config backup or ios_config backup and stores snapshots somewhere (a separate git repo, S3 bucket, or network management tool like Oxidized/RANCID)
- This is for disaster recovery and drift detection, not for editing
- You never edit these backups directly — changes always flow through the intent YAML

## The Flow at Scale

You edit: configs/roles/leaf.yml  (change NTP servers for all leaves)
           or
          configs/devices/dc1-leaf-01.yml  (change one device's BGP peer)
       |
Git diff is small, reviewable, meaningful
       |
Ansible playbook merges global + role + site + device YAML
       |
Resource modules (nxos_vlans, nxos_bgp_global, ios_ntp_global, etc.)
compute the delta between desired state and live device state
       |
Only the actual changes get pushed to the device

## So to Directly Answer Your Question

Intent/desired state YAML      -> Stored in Git (Dev + Prod repos)
Full running configs           -> NOT in Git (Backup system: Oxidized, RANCID, S3, separate repo)
Jinja2 templates (if used)     -> Stored in Git (Same repos)
Secrets/credentials            -> NOT in Git (AAP credential store or vault)

The YAML in git is your source of truth for what the network should look like, not a mirror of what's currently running. Ansible's resource modules handle the translation from intent to device commands.
